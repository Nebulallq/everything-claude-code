# 单 Agent + 人工确认流程

## —— Human-in-the-Loop 自动化研发协议

------

# 🎯 目标

在自动化开发前，必须经过人工确认： 

1. 模型读取需求
2. 输出结构化理解结果
3. 人工确认无误
4. 再进入设计与开发阶段

避免：

- 需求理解偏差
- 错误方向实现
- 无效代码修改
- 浪费测试成本

------

# 🏗 总体执行流程

```text
需求输入
   ↓
阶段1：需求理解复述（模型输出）
   ↓
人工确认（必须）
   ↓
阶段2：技术设计
   ↓
阶段3：代码实现
   ↓
阶段4：测试补充
   ↓
阶段5：综合自检与验证
   ↓
阶段6：更新模块功能文档
   ↓
完成
```

⚠ 未经确认，不得进入设计阶段
⚠ 不允许跳过确认

------

# 🧠 核心原则

## 1️⃣ 强制理解锁

模型必须输出：

- 功能目标
- 业务规则
- 边界条件
- 异常场景
- 不明确点（主动提问）

格式必须结构化。

------

## 2️⃣ 不做隐式假设原则

若需求不明确：

- 必须列出疑问
- 不得自行假设实现
- 等待人工补充说明

------

## 3️⃣ 人工确认门

进入设计阶段的唯一条件：

- 人工明确回复：“确认无误，进入设计阶段”

否则停止执行。

# 📌 阶段2：技术设计（含输出模板）

设计必须：

- 围绕最小变更
- 明确修改类
- 明确新增方法
- 明确是否涉及 DB
- 明确是否涉及接口变更

禁止：

- 输出泛泛架构论文
- 长篇理论分析

---

## 阶段2输出模板

```markdown
# 技术设计方案

## 一、设计方案
- 核心改动：
- 涉及文件：
- 新增/修改类/方法：

## 二、风险评估
| 等级 | 风险项 | 应对措施 |
|-----|-------|---------|
| HIGH | [严重风险] | [解决方案] |
| MEDIUM | [中等风险] | [应对措施] |
| LOW | [轻微风险] | 记录备案 |

**风险等级定义**:
- HIGH：可能阻塞实现的严重风险，必须设计解决方案
- MEDIUM：中等风险，应说明应对措施
- LOW：轻微风险，可接受，记录备案

**风险类别**:
- 技术依赖风险（外部 API、第三方库）
- 兼容性风险（破坏性变更、版本冲突）
- 性能风险（数据量、并发场景）
- 安全风险（敏感数据、权限控制）

## 三、依赖检查
- 依赖库版本：
- 数据库变更：
- 配置变更：
- API 变更：

## 四、兼容性影响
- 破坏性变更：
- 向后兼容性：
- 回滚方案：

请确认设计无误，进入实现阶段。
```

# 📌 阶段4：测试补充（含 TDD 执行流程）

## TDD 执行循环

```
RED → GREEN → REFACTOR → REPEAT
```

### 执行步骤
1. **SCAFFOLD**: 定义接口、函数签名（抛出 "Not implemented"）
2. **RED**: 编写失败的测试（必须先验证测试失败）
3. **GREEN**: 编写最小可行代码使测试通过
4. **REFACTOR**: 重构改进代码质量
5. **COVERAGE**: 检查覆盖率是否达标

## 覆盖率要求

| 代码类型 | 最低覆盖率 |
|---------|-----------|
| 标准代码 | 80% |
| 财务计算 | 100% |
| 认证逻辑 | 100% |
| 安全关键代码 | 100% |

## 必须新增的测试类型

- **单元测试**: 单个函数、组件
- **边界测试**: 空值、null、最大值、边界条件
- **错误测试**: 无效输入、网络失败、异常场景
- **集成测试**: API 端点、数据库操作

## 阶段4输出模板

```markdown
# 测试补充报告

## 一、测试覆盖情况
| 测试类型 | 测试数量 | 覆盖率 |
|---------|---------|--------|
| 单元测试 | X | XX% |
| 集成测试 | X | XX% |
| E2E测试 | X | - |

## 二、测试清单
- [ ] 正常流程测试
- [ ] 异常流程测试
- [ ] 边界条件测试
- [ ] 错误处理测试
- [ ] 覆盖率 ≥ 80%

## 三、测试执行结果
```

# 📌 阶段5：综合自检与验证

## 自检清单

### 安全问题（CRITICAL - 必须修复）
- [ ] 无硬编码凭证（API keys、密码、tokens）
- [ ] 无 SQL 注入风险（使用参数化查询）
- [ ] 无 XSS 漏洞（输出编码、CSP）
- [ ] 输入验证完整

**OWASP Top 10 检查**:
1. [ ] 注入攻击：参数化查询、输入清理
2. [ ] 身份认证：密码存储（bcrypt/argon2）、会话管理
3. [ ] 敏感数据：加密存储/传输、密钥管理
4. [ ] XXE：禁用 DTD 处理
5. [ ] 访问控制：授权检查、角色验证
6. [ ] 安全配置：移除默认凭证、错误处理不泄露信息
7. [ ] XSS：输出编码、内容安全策略
8. [ ] 反序列化：数据验证、完整性检查
9. [ ] 已知漏洞：运行 `npm audit` / 依赖更新
10. [ ] 日志监控：安全事件记录、无敏感数据

### 代码质量（HIGH - 强烈建议修复）
- [ ] 函数不超过 50 行
- [ ] 文件不超过 800 行
- [ ] 嵌套深度不超过 4 层
- [ ] 无遗漏的错误处理
- [ ] 无 console.log 调试语句
- [ ] 公共 API 有 JSDoc 注释

### 最佳实践（MEDIUM - 建议修复）
- [ ] 使用不可变模式
- [ ] 无过度复杂的逻辑
- [ ] 新代码有测试覆盖
- [ ] 无 TODO/FIXME 注释

### 原有检查项
- [ ] 是否存在逻辑漏洞
- [ ] 是否可能 NPE
- [ ] 是否违反原有分层
- [ ] 是否破坏旧接口

## 综合验证流程

```bash
# 1. 构建验证
npm run build

# 2. 类型检查
npx tsc --noEmit

# 3. 代码检查
npm run lint

# 4. 测试套件
npm test -- --coverage

# 5. 安全扫描
# 检查硬编码凭证
grep -rn "sk-\|api_key\|password" --include="*.ts" --include="*.js" . | head -10
# 检查 console.log
grep -rn "console.log" --include="*.ts" src/ | head -10
```

## 验证报告格式

```
VERIFICATION REPORT
==================
Build:     [PASS/FAIL]
Types:     [PASS/FAIL] (X errors)
Lint:      [PASS/FAIL] (X warnings)
Tests:     [PASS/FAIL] (X/Y passed, Z% coverage)
Security:  [PASS/FAIL] (X issues)

Overall:   [READY/NOT READY] for PR

Issues to Fix:
1. [具体问题]
2. [具体问题]
```

# 📌 阶段6：更新模块功能文档

## 目的

开发完成后，**更新**对应模块的功能文档，保持文档与代码同步。

## 文档结构

### 📁 目录结构
```
docs/
  modules/
    user-auth.md       # 用户认证模块
    payment.md         # 支付模块
    data-export.md     # 数据导出模块
    notification.md    # 通知模块
    ...
```

## 文档模板

每个模块文档统一使用以下结构：

**文件头部**
```
# [模块名称]

## 模块概述
[模块的整体描述和定位]

## 功能列表
```

**每个功能使用以下格式重复**
```
### [功能名称]

**功能描述**
[清晰描述这个功能做什么]

**使用示例**
[代码示例]

**使用限制**
- [如果有限制或约束，否则此项可省略]

---
```

## 执行规则

1. **识别所属模块**：根据改动内容判断属于哪个功能模块
2. **查找文档**：
   - 存在：更新对应功能的描述
   - 不存在：创建新文档 `docs/modules/[module-name].md`
3. **更新内容**：
   - **新增功能**：添加新功能章节
   - **修改功能**：更新对应功能的描述和示例
   - **删除功能**：删除对应功能章节

# 🔁 错误处理流程

## 理解偏差修正

1. 人工指出理解错误的具体点
2. Agent 重新输出修正后的理解
3. 人工再次确认
4. 最多重试 3 次，超过则转为人工介入

## 设计阶段回退

- 若设计中发现需求理解有误，立即回退到阶段1
- 保持已确认的部分不变，只修正有误部分

## 实现阶段问题处理

- 若实现中发现设计有误，评估影响范围
- 小问题直接修正，大问题回退到阶段2
- 记录所有变更和原因

# 📋 人工确认规范

## 标准确认回复格式

```
确认无误，进入XX阶段
```

## 有条件确认

```
确认，但需补充：XXX
```

## 拒绝确认（必须说明原因）

```
拒绝，原因：XXX
```

## 需求变更处理

| 变更时机 | 处理方式 |
|---------|---------|
| 阶段1确认前 | 直接修改需求，重新输出理解 |
| 阶段2设计后 | 评估变更影响，可能需要重新设计 |
| 阶段3实现中 | 评估已完成工作的复用性，可能需要回退 |

Agent 必须输出：
- 已完成工作的复用性
- 额外工作量估算
- 是否需要回退

# 🎯 适用边界

## 适合的场景

- 功能明确的需求
- 单人可完成的改动
- 非核心业务流程
- 测试充分的模块
- 中小型项目（1-5人团队）

## 不适合的场景

- 需求模糊且无法澄清
- 涉及多个团队协作
- 核心架构重构
- 高风险性能优化
- 需要深度业务理解的功能

# 🤖 Agent 能力声明

## 能做到

- 读取和理解现有代码
- 遵循项目代码风格
- 生成符合规范的测试
- 识别常见的代码问题
- 执行结构化的检查清单
- 输出标准化的文档和报告

## 做不到

- 理解隐含的业务逻辑
- 处理极其模糊的需求
- 替代人工的架构决策
- 保证 100% 无缺陷
- 处理跨团队的复杂协调

# 📊 为什么推荐这种模式？

相比完全自动化：

| 模式              | 稳定性 | 可控性 | 误解风险 |
| ----------------- | ------ | ------ | -------- |
| 完全无人          | 中     | 低     | 高       |
| Human-in-the-Loop | 高     | 高     | 低       |

------

# 🧭 总结

自动化不等于无人化。

真正成熟的工程自动化模式是：

> 机器做理解
> 人类做确认
> 机器做执行

这种结构：

- 成功率更高
- 风险更低
- 更适合企业级研发流程
